    Лабораторная работа №2 — Проектирование и реализация пользовательской формы с UX-валидацией

    Цель

    Создать интерактивную UX-ориентированную форму по индивидуальной предметной области (сервис №1–30), реализовав:

    • продуманный User Flow;
    • клиентскую валидацию на JavaScript;
    • inline-ошибки и подсказки;
    • маски ввода (телефон/дата/время);
    • визуальные состояния полей (focus / success / error);
    • доступность (A11y) через aria-invalid, aria-describedby;
    • UX-поведение после отправки (toast/alert/confirmation block);
    • публикацию результата в GitHub с ссылкой в отчёте.

    1. Теория

    1.1. UX-форма и отличие от “просто формы”

    Обычная форма часто:

    • проверяет только на submit;
    • показывает ошибку “где-то сверху”;
    • не объясняет, как исправить.

    UX-форма:

    • предупреждает ошибки заранее (на blur/input);
    • показывает inline-ошибки рядом с полем;
    • даёт подсказки формата (placeholder/help);
    • ведёт пользователя к успешной отправке;
    • дружит с клавиатурой и скринридером.

    Пример UX-поведения (что пользователь видит)

    • Ввёл телефон “8999” → после ухода из поля видит: “Введите 11 цифр, формат +7 (999) 111-22-33”.
    • Исправил → рамка зелёная, ошибка исчезла.
    • Нажал “Отправить” → при ошибке фокус прыгает в первое проблемное поле.

    Мини-пример (структура поля с подсказкой и ошибкой)

    <div class="field">
  <label for="fullname">ФИО</label>
  <input id="fullname" name="fullname" type="text"
         autocomplete="name"
         aria-describedby="fullname-help fullname-error"
         aria-invalid="false" required>
  <small id="fullname-help">Введите полностью: Фамилия Имя Отчество</small>
  <small id="fullname-error" class="error" role="alert"></small>
</div>

    1.2. Выбор предметной области и сценария формы

    Каждый студент выбирает один сервис (№1–30) и делает ключевой сценарий формы.

    Как правильно определить “ключевой сценарий”

    Ключевой сценарий = действие, ради которого пользователь пришёл в сервис:

    • №1 переговорные → бронирование
    • №2 врач → запись на приём
    • №4 доставка → оформление заказа
    • №20 service portal → создание заявки

    Пример фиксации в отчёте (как писать)

    Предметная область: №20 Портал подачи технических заявок (Service Portal)
Сценарий формы: создание заявки в поддержку
Цель формы: быстро зарегистрировать обращение и получить номер заявки
Пользователь: сотрудник компании (инициатор заявки)

    1.3. User Flow: структура формы и логика взаимодействия

    User Flow — это “маршрут” пользователя: шаги + реакции системы. Минимум 4 состояния обязателен.

    Пример User Flow (универсальный, 4 состояния)

    Старт → Заполнение → Проверка/Ошибки → Успех/Подтверждение

    Пример User Flow для “заявки в поддержку” (с зависимостью)

    Старт формы →
Выбор категории →
(Если категория = "Доступ") появляется поле "Логин" (обязательное) →
Заполнение контактов →
Submit →
Ошибки? да → inline + фокус →
Ошибок нет → toast + номер заявки

    Пример оформления flow в HTML (если делаете “псевдо-экраны” секциями)

    <section id="step-1" aria-label="Шаг 1">
  <h2>Шаг 1 — Тема и категория</h2>
</section>

<section id="step-2" aria-label="Шаг 2">
  <h2>Шаг 2 — Контакты</h2>
</section>

<section id="step-3" aria-label="Шаг 3">
  <h2>Шаг 3 — Подтверждение</h2>
</section>

    1.4. Структура данных формы: “поле → тип → правило → ошибка”

    Перед кодом нужно зафиксировать структуру: какие поля, какие правила.

    Пример таблицы полей (фрагмент)

    ФИО → text → required, minlength=5 → "Введите ФИО (минимум 5 символов)"
Телефон → tel → required, pattern → "Телефон в формате +7 (999) 111-22-33"
Дата → text → required, custom date → "Введите дату ДД.ММ.ГГГГ"
Способ → radio → required → "Выберите вариант"
Комментарий → textarea → minlength=20 → "Минимум 20 символов"
Согласие → checkbox → required → "Подтвердите согласие"

    Пример “карты правил” в JS (как хранить требования)

    const rules = {
  fullname: { required: true, minLength: 5, message: "Введите ФИО (≥ 5 символов)" },
  phone:    { required: true, pattern: /^\+7 \(\d{3}\) \d{3}-\d{2}-\d{2}$/, message: "Формат: +7 (999) 111-22-33" },
  date:     { required: true, pattern: /^\d{2}\.\d{2}\.\d{4}$/, message: "Формат: ДД.ММ.ГГГГ" }
};

    1.5. Верстка формы: семантика + форма + подсказки/ошибки

    Требование: в index.html должны быть семантические блоки и правильная структура формы.

    Пример “каркаса страницы” под лабораторную

    <header id="top">
  <h1>Сервис бронирования / заявок / регистрации</h1>
  <p>Краткое описание предметной области (1–2 предложения).</p>
</header>

<nav aria-label="Навигация">
  <ul>
    <li><a href="#form">Форма</a></li>
    <li><a href="#rules">Правила</a></li>
    <li><a href="#contacts">Контакты</a></li>
  </ul>
</nav>

<main id="main">
  <section id="form">
    <h2>Форма</h2>

    <form id="uxForm" novalidate>
      <fieldset>
        <legend>Данные пользователя</legend>
        <!-- поля -->
      </fieldset>

      <fieldset>
        <legend>Данные заявки/заказа</legend>
        <!-- поля -->
      </fieldset>

      <button type="submit">Отправить</button>
      <button type="reset">Очистить</button>
    </form>
  </section>

  <aside>
    <h2>Подсказки</h2>
    <p><strong>Важно:</strong> поля со * обязательны.</p>
  </aside>
</main>

<footer id="contacts">
  <address>
    Email: <a href="mailto:support@example.com">support@example.com</a><br>
    Телефон: <a href="tel:+79990001122">+7 (999) 000-11-22</a>
  </address>
</footer>

    1.6. Inline-ошибки: как правильно выводить

    Inline-ошибка — текст рядом с полем. Он должен:

    • появляться после blur или после submit;
    • исчезать после исправления;
    • быть конкретным.

    Пример поля с inline-ошибкой (HTML)

    <div class="field">
  <label for="email">Email</label>
  <input id="email" name="email" type="email"
         aria-describedby="email-help email-error"
         aria-invalid="false"
         autocomplete="email" required>
  <small id="email-help">Пример: user@mail.com</small>
  <small id="email-error" class="error" role="alert"></small>
</div>

    Пример функции вывода ошибки (JS)

    function setError(input, message) {
  const error = document.getElementById(`${input.id}-error`);
  input.classList.add("is-invalid");
  input.classList.remove("is-valid");
  input.setAttribute("aria-invalid", "true");
  error.textContent = message;
}

function setSuccess(input) {
  const error = document.getElementById(`${input.id}-error`);
  input.classList.add("is-valid");
  input.classList.remove("is-invalid");
  input.setAttribute("aria-invalid", "false");
  error.textContent = "";
}

    1.7. Маски ввода: телефон/дата/время

    Маска делает формат ввода “безошибочным”.

    Пример HTML для масок

    <input id="phone" type="tel" inputmode="tel"
       placeholder="+7 (___) ___-__-__"
       aria-describedby="phone-error"
       aria-invalid="false" required>
<small id="phone-error" class="error" role="alert"></small>

<input id="date" type="text" inputmode="numeric"
       placeholder="ДД.ММ.ГГГГ"
       aria-describedby="date-error"
       aria-invalid="false" required>
<small id="date-error" class="error" role="alert"></small>

    Пример простой маски телефона (JS, минимально достаточно для ЛР)

    function maskPhone(value) {
  const digits = value.replace(/\D/g, "").replace(/^8/, "7");
  const d = digits.startsWith("7") ? digits : ("7" + digits);

  const p1 = d.slice(1, 4);
  const p2 = d.slice(4, 7);
  const p3 = d.slice(7, 9);
  const p4 = d.slice(9, 11);

  let out = "+7";
  if (p1) out += ` (${p1}`;
  if (p1.length === 3) out += ")";
  if (p2) out += ` ${p2}`;
  if (p3) out += `-${p3}`;
  if (p4) out += `-${p4}`;
  return out;
}

document.getElementById("phone").addEventListener("input", (e) => {
  e.target.value = maskPhone(e.target.value);
});

    Пример простой маски даты (JS)

    function maskDate(value) {
  const d = value.replace(/\D/g, "").slice(0, 8);
  const p1 = d.slice(0, 2);
  const p2 = d.slice(2, 4);
  const p3 = d.slice(4, 8);
  return [p1, p2, p3].filter(Boolean).join(".");
}

document.getElementById("date").addEventListener("input", (e) => {
  e.target.value = maskDate(e.target.value);
});

    1.8. Визуальные состояния: focus / valid / invalid

    Нужно, чтобы пользователь видел состояние поля.

    Пример CSS

    .field input, .field select, .field textarea {
  border: 1px solid #999;
  padding: 10px;
  border-radius: 8px;
}

.field input:focus, .field select:focus, .field textarea:focus {
  outline: 2px solid #000;
  outline-offset: 2px;
}

.is-valid { border: 2px solid green !important; }
.is-invalid { border: 2px solid red !important; }

.error { color: red; min-height: 1em; display: block; }

    Пример применения классов из валидации (JS)

    if (isOk) setSuccess(input);
else setError(input, "Сообщение ошибки");

    1.9. A11y: label/aria-invalid/aria-describedby/role=alert

    Минимальная доступность:

    1. label for привязан к id
    2. aria-describedby содержит id подсказки/ошибки
    3. при ошибке aria-invalid="true"
    4. блок ошибки role="alert"

    Пример правильной связки

    <label for="comment">Комментарий</label>
<textarea id="comment" name="comment"
          aria-describedby="comment-help comment-error"
          aria-invalid="false"
          minlength="20" required></textarea>
<small id="comment-help">Опишите ситуацию минимум в 20 символов</small>
<small id="comment-error" class="error" role="alert"></small>

    1.10. Зависимость полей: минимум одна

    Зависимость = одно поле управляет другим (видимость, required, список вариантов, стоимость).

    Пример: доставка курьером → адрес обязателен (HTML)

    <fieldset>
  <legend>Способ получения</legend>

  <label><input type="radio" name="delivery" value="courier" required> Курьер</label>
  <label><input type="radio" name="delivery" value="pickup" required> Самовывоз</label>
  <small id="delivery-error" class="error" role="alert"></small>
</fieldset>

<div class="field" id="addressBlock" hidden>
  <label for="address">Адрес</label>
  <input id="address" name="address" type="text"
         aria-describedby="address-error"
         aria-invalid="false"
         autocomplete="street-address">
  <small id="address-error" class="error" role="alert"></small>
</div>

    Пример реализации зависимости (JS)

    const addressBlock = document.getElementById("addressBlock");
const addressInput = document.getElementById("address");

document.querySelectorAll('input[name="delivery"]').forEach(r => {
  r.addEventListener("change", () => {
    const isCourier = document.querySelector('input[name="delivery"]:checked')?.value === "courier";
    addressBlock.hidden = !isCourier;

    if (isCourier) {
      addressInput.setAttribute("required", "required");
    } else {
      addressInput.removeAttribute("required");
      addressInput.value = "";
      addressInput.classList.remove("is-valid", "is-invalid");
      addressInput.setAttribute("aria-invalid", "false");
      document.getElementById("address-error").textContent = "";
    }
  });
});

    1.11. UX-поведение при submit: запрет, фокус, toast/confirmation

    Требуется:

    • при ошибках — не отправлять;
    • поставить фокус на первое ошибочное поле;
    • при успехе — показать toast или confirmation.

    Пример confirmation block (HTML)

    <div id="result" class="result" hidden>
  <h2>Готово</h2>
  <p>Заявка принята. Номер: <span id="ticketId"></span></p>
  <a href="#top">Наверх</a>
</div>

    Пример toast (HTML)

    <div id="toast" class="toast" hidden role="status" aria-live="polite">
  Данные отправлены успешно
</div>

    Пример submit-логики (JS)

    const form = document.getElementById("uxForm");
const toast = document.getElementById("toast");
const result = document.getElementById("result");

form.addEventListener("submit", (e) => {
  e.preventDefault();

  const inputs = form.querySelectorAll("input, select, textarea");
  let firstInvalid = null;

  inputs.forEach(el => {
    const ok = el.checkValidity();
    if (!ok) {
      if (!firstInvalid) firstInvalid = el;
      setError(el, el.validationMessage || "Проверьте поле");
    } else {
      // не всегда ставим success на пустые необязательные — по ситуации
      if (el.value.trim() !== "" || el.hasAttribute("required")) setSuccess(el);
    }
  });

  if (firstInvalid) {
    firstInvalid.focus();
    return;
  }

  // успех
  const id = Math.floor(1000 + Math.random() * 9000);
  const ticket = document.getElementById("ticketId");
  if (ticket) ticket.textContent = `#${id}`;

  if (toast) {
    toast.hidden = false;
    setTimeout(() => (toast.hidden = true), 2500);
  }
  if (result) result.hidden = false;

  // опционально: очистка
  // form.reset();
});

    1.12. Тестирование сценариев: минимум 3 тест-кейса

    Тест-кейс = входные данные + ожидаемый результат.

    Тест-кейс 1: пустая форма

    Шаги: нажать "Отправить" при пустых полях
Ожидаемо: форма не отправляется; под обязательными полями inline-ошибки; фокус на первом обязательном

    Тест-кейс 2: неверный формат телефона/даты

    Данные: телефон "123", дата "99.99.9999"
Ожидаемо: ошибки формата под соответствующими полями; aria-invalid=true; подсветка красная

    Тест-кейс 3: корректные данные

    Данные: заполнены все обязательные поля корректно
Ожидаемо: ошибок нет; появляется toast/confirmation; (опционально) присваивается номер заявки

    2. Задания

    Каждый студент выбирает свою предметную область (сервис №1–30) и делает форму ключевого сценария.

    2.1. Выбор предметной области и сценария формы

    1. Выберите номер сервиса (1–30).
    2. Определите главный сценарий формы (пример: бронь/заявка/регистрация/заказ/перевод).

    3. Запишите в отчёте:

    4. сервис № и название;

    5. цель формы (1–2 предложения);
    6. кто пользователь (роль).

    2.2. Проектирование User Flow

    Сделайте User Flow минимум на 4 состояния:

    • старт формы;
    • заполнение;
    • ошибки/исправления;
    • успех/подтверждение.

    Формат в отчёте:

    • схема со стрелками или текстовый flow.

    2.3. Проектирование структуры данных формы

    Составьте таблицу “Поле → тип → правило → сообщение ошибки”.

    Минимум 8 полей, из них обязательно:

    • 2 поля с маской;
    • 1 зависимость;
    • 1 select или radio;
    • 1 textarea или input type="file";
    • чекбокс согласия.

    2.4. Верстка формы

    Сделайте index.html:

    • семантика: header, main, section, footer;
    • форма: form, fieldset, legend, label, input, select, textarea;

    • для каждого поля:

    • id, name, label for;

    • блок подсказки (help);
    • блок ошибки (error).

    2.5. Реализация inline-ошибок

    Требования:

    • ошибки выводятся под полем;
    • при исправлении ошибки — исчезает;
    • текст ошибки — конкретный.

    2.6. Реализация масок ввода

    Сделайте минимум 2 маски, варианты:

    • телефон + дата,
    • телефон + время,
    • дата + время.

    2.7. Реализация визуальных состояний

    Реализовать стили:

    • focus,
    • is-valid,
    • is-invalid.

    2.8. Реализация A11y

    Требования:

    • label for у всех полей;
    • при ошибке: aria-invalid="true";
    • aria-describedby связывает поле с текстом ошибки/подсказки;
    • блок ошибок имеет role="alert".

    2.9. Реализация зависимости полей

    Примеры зависимостей на выбор (минимум одна):

    • тариф → стоимость;
    • способ доставки → адрес обязателен;
    • выбор даты → доступное время;
    • тип заявки → доп.поле появляется и становится required.

    2.10. UX-поведение при submit

    На отправке:

    • запрещать отправку при ошибках;
    • переводить фокус на первое ошибочное поле;
    • при успехе показывать toast/confirmation;
    • опционально: очистить форму или сохранить “черновик”.

    2.11. Тестирование сценариев

    В отчёте описать минимум 3 тест-кейса:

    1. пустая форма → ошибки;
    2. неверный формат телефона/даты → ошибка;
    3. корректные данные → успех.

    • Репозиторий GitHub.
    • Скриншот проверки (валидатор HTML или скрин консоли без ошибок).
