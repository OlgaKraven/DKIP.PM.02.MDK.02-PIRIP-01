    Лабораторная работа №6 — Оптимизация графики и визуальной производительности интерфейса

    Цель

    Оптимизировать графику и визуальную производительность интерфейса по индивидуальной предметной области (сервис №1–30), реализовав:

    • изображения WebP + fallback;
    • SVG-иконки (inline или sprite);
    • lazy loading для изображений;
    • responsive images (srcset, sizes);
    • минимизацию размеров (качество/разрешение/вес);
    • оптимизацию hero-блока (LCP);
    • анализ Lighthouse Performance и фиксацию улучшений.

    1. Теория

    1.1. Почему “графика” влияет на производительность UI

    Основные метрики, которые страдают из-за картинок:

    • LCP (Largest Contentful Paint) — часто это hero-картинка/баннер.
    • CLS (Cumulative Layout Shift) — скачки из-за отсутствия размеров у <img>.
    • TBT/INP — косвенно (тяжелый CSS/JS, но картинки тоже влияют через сеть).

    Главная идея: показываем быстро то, что видно сразу, а остальное грузим позже и легче.

    1.2. WebP + fallback: современный формат без потерь качества

    WebP обычно даёт меньший размер файла при сопоставимом качестве. Fallback нужен для совместимости (если браузер не поддерживает WebP или вы даёте запасной формат).

    Пример: <picture> WebP + JPEG fallback

    <picture>
  <source srcset="assets/hero.webp" type="image/webp">
  <img src="assets/hero.jpg" width="1200" height="600"
       alt="Hero-блок сервиса" decoding="async">
</picture>

    Важно для CLS: задавайте width и height.

    1.3. SVG-иконки: sprite или inline

    SVG лучше для UI-иконок:

    • масштабируется без пикселей,
    • небольшой вес,
    • можно менять цвет через CSS.

    Вариант A: inline SVG (быстро и просто)

    <button class="btn">
  <svg width="20" height="20" viewBox="0 0 24 24" aria-hidden="true">
    <path d="M3 12h18" />
    <path d="M12 3v18" />
  </svg>
  Добавить
</button>

    Вариант B: SVG sprite (один файл на все иконки)

    sprite.svg

    <svg xmlns="http://www.w3.org/2000/svg" style="display:none">
  <symbol id="i-search" viewBox="0 0 24 24">
    <path d="M10 18a8 8 0 1 1 0-16a8 8 0 0 1 0 16Z" />
    <path d="M21 21l-4.3-4.3" />
  </symbol>
</svg>

    использование

    <svg width="20" height="20" aria-hidden="true">
  <use href="#i-search"></use>
</svg>

    1.4. Lazy loading изображений

    Lazy loading = грузить картинки когда они попадают в область видимости.

    Пример: нативный lazy loading

    <img src="assets/card-1.webp"
     alt="Карточка"
     loading="lazy"
     decoding="async"
     width="480" height="320">
    • loading="lazy" — откладывает загрузку.
    • decoding="async" — браузер декодирует без блокировки.

    1.5. Responsive images: srcset + sizes

    Одна и та же картинка должна грузиться разного размера под разные экраны.

    Пример: srcset + sizes

    <img
  src="assets/product-800.webp"
  srcset="
    assets/product-400.webp 400w,
    assets/product-800.webp 800w,
    assets/product-1200.webp 1200w
  "
  sizes="(max-width: 600px) 90vw, (max-width: 1024px) 45vw, 30vw"
  alt="Изображение товара"
  loading="lazy"
  decoding="async"
  width="800" height="600">

    Смысл:

    • srcset — какие варианты есть,
    • sizes — сколько места займёт на разных ширинах экрана.

    1.6. Минимизация размеров изображений (качество, размеры, вес)

    Оптимизация изображения включает:

    • уменьшение пикселей (не грузить 4000px, если нужно 800px);
    • подбор качества (например, WebP quality 70–85);
    • удаление метаданных (EXIF);
    • выбор правильного формата (фото → WebP/JPEG, графика → SVG/PNG/WebP).

    Пример требований (в отчёт)

    До: hero.jpg 2400×1200, 1.8 MB
После: hero.webp 1600×800, 220 KB

    1.7. Оптимизация hero-блока (LCP)

    Hero — главный кандидат на LCP. Цель: чтобы он загрузился первым и быстро.

    Что делать:

    • hero изображение — не lazy (оно above the fold);
    • указать размеры;
    • возможно fetchpriority="high" (если нужно ускорить LCP).

    Пример hero с приоритетом

    <picture>
  <source srcset="assets/hero-1600.webp" type="image/webp">
  <img src="assets/hero-1600.jpg"
       alt="Главный баннер"
       width="1600" height="800"
       decoding="async"
       fetchpriority="high">
</picture>

    1.8. Lighthouse Performance: как анализировать

    В Lighthouse фиксируем:

    • Performance score;
    • LCP (время);
    • объём переданных данных (transfer size);
    • рекомендации (properly size images, next-gen formats, defer offscreen images).

    Что обязательно зафиксировать в отчёте

    1. Скрин “до”.
    2. Скрин “после”.
    3. Таблица улучшений (минимум 3 пункта).

    2. Задания

    Предметная область — сервис №1–30. Оптимизация выполняется на странице интерфейса (каталог/дашборд/сервис).

    2.1. Подготовка проекта

    Структура:

    lr6/
  index.html
  assets/
    img/
    icons/
  report/

    2.2. WebP + fallback

    1. Подготовить минимум 3 изображения в WebP.
    2. Для каждого — обеспечить fallback через <picture>.

    2.3. SVG-иконки

    Использовать минимум 5 иконок:

    • sprite или inline — на выбор,
    • цвета должны управляться через CSS (fill/stroke).

    2.4. Lazy loading

    1. Все изображения ниже первого экрана — loading="lazy".
    2. Добавить decoding="async".

    2.5. Responsive images

    Минимум 2 изображения должны иметь:

    • srcset (минимум 3 размера),
    • sizes.

    2.6. Минимизация размеров изображений

    Сделать таблицу “до/после” минимум на 3 изображения:

    • размеры в px,
    • вес в KB/MB.

    2.7. Оптимизация hero-блока

    Hero-блок должен:

    • иметь заданные width/height;
    • грузиться приоритетно (не lazy);
    • быть в WebP + fallback.

    2.8. Lighthouse Performance

    1. Запустить Lighthouse (Performance) “до”.
    2. Внести оптимизации.
    3. Запустить Lighthouse “после”.

    4. Приложить скрины и краткий вывод:

    5. что улучшилось;

    6. какие 2 рекомендации Lighthouse исчезли.

    • GitHub-репозиторий.
    • Ссылка в отчёте.
    • При необходимости — GitHub Pages.
